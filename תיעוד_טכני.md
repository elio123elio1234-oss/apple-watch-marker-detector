## 1. סקירת הפרויקט

**מטרה:** פיתוח מערכת סמנים פידוציאליים המוצגת על מסך Apple Watch, המאפשרת למצלמת מחשב נייד לזהות במדויק את המיקום התלת-ממדי, הכיוון וזווית הסיבוב של השעון ביחס לגוף בזמן אמת. המטרה העיקרית היא מעקב אחר מיקומי אלקטרודות ECG (V1-V6) במהלך הליכים רפואיים.

**האתגר:** סמנים פידוציאליים מסורתיים (קודי QR, סמני ArUco) נכשלים על מסכי OLED עקב:
- מבנה ברמת הפיקסל הנראה מטווח קרוב
- תופעת bloom ב-OLED (פיקסלים בהירים "דולפים" לשכנים)
- השתקפויות ספקולריות על זכוכית השעון
- זוויות צפייה קיצוניות (הטיה של 70-80° במיקומי V5/V6)

**הפתרון:** סמן מלא-מסך דו-צבעי (SplitScreen) הממלא את כל תצוגת ה-Apple Watch בשני צבעים נבדלים (ציאן וירוק), ומספק פי 100-500 יותר שטח ניתן לזיהוי לעומת סמנים מבוססי נקודות מסורתיים.

## 2. אבולוציית העיצוב

### שלב 1: סמן QuadDot (גרסאות 1-3)
**קונצפט:** ארבע נקודות עגולות לבנות מסודרות בצורת מרובע, עם נקודה אחת גדולה יותר לכיוון.

**בעיות שהתגלו:**
1. **חיוביים שקריים (False Positives):** רהיטים לבנים, ניירות וביגוד גרמו לזיהויים מוטעים
2. **תופעת Bloom ב-OLED:** בבהירות 100%, נקודות לבנות "דלפו" זו לזו, עיוותו את הגאומטריה
3. **כשל בזוויות קיצוניות:** בהטיה של 70-75° (מיקום V5/V6), כל נקודה התכווצה ל-3-5 פיקסלים ולא ניתנה לזיהוי

**ניסיונות תיקון:**
- **גרסה 2:** מעבר לנקודות ירוקות עם סינון צבע HSV ואימות רקע כהה
- **גרסה 3:** הפחתת בהירות ל-60%, התאמת אליפסות לסובלנות bloom
- **תוצאה:** שיפור אך עדיין כשל בזוויות תלולות עקב שטח פיקסלים לא מספק

### שלב 2: סמן SplitScreen (גרסה 4 - סופי)
**התובנה פורצת הדרך:** בזוויות קיצוניות, זיהוי 4 נקודות קטנות הוא בלתי אפשרי, אבל זיהוי מלבן צבעוני גדול הוא טריוויאלי.

**עיצוב:**
- **חצי שמאלי:** ציאן (HSV: גוון 78-115°, רוויה 70+, ערך 25+)
- **חצי ימני:** ירוק (HSV: גוון 25-76°, רוויה 70+, ערך 25+)
- **בהירות:** 60% (מפחית bloom תוך שמירה על יכולת זיהוי)
- **גאומטריה:** ממלא את כל מסך השעון (37.6×46.0 מ"מ עבור דגם 45mm)

**למה זה עובד:**
- במיקום V6 (הטיה 75°, מרחק 40 ס"מ), המסך מוקרן ל-~20×80 פיקסלים
- QuadDot: 4 נקודות × 3px כל אחת = 12 פיקסלים סה"כ
- SplitScreen: ~20×80 = 1600 פיקסלים (פי 133 יותר שטח)
- הפיצול הצבעוני מספק כיוון גם כשהתמונה דחוסה מאוד

## 3. רציונל בחירת הצבעים

### למה ציאן + ירוק?

**תכונות קולורימטריות:**
- **הפרדה ספקטרלית:** ציאן (480-490nm) וירוק (520-560nm) מופרדים מספיק בספקטרום הנראה כדי להישאר מובחנים תחת:
  - תאורה סביבתית משתנה (ליבון, פלורסנט, LED, אור יום)
  - שינויי איזון לבן של המצלמה
  - השתקפויות זכוכית (שמוסיפות אור לבן, מפחיתות רוויה אך משמרות גוון)

- **חוסן HSV:** מרחב הצבעים HSV (Hue-Saturation-Value) חסין יותר לשינויי תאורה מ-RGB:
  - **גוון (Hue):** יציב תחת שינויי בהירות
  - **רוויה (Saturation):** מבדיל עצמים צבעוניים ממשטחים לבנים/אפורים
  - **ערך (Value):** מאפשר זיהוי בטווח בהירות רחב (25-255)

**למה לא שילובי צבעים אחרים?**
- **אדום + כחול:** אדום בעל רגישות מצלמה גרועה; כחול נדיר בגווני עור אך חופף לג'ינס/ביגוד
- **צהוב + מגנטה:** דומים מדי בגוון תחת תאורה פלורסנטית
- **לבן + שחור:** אין מידע צבע; שיעור גבוה של false positives (כל חפץ מלבני)

### פרמטרי מסנן HSV

```python
WIDE_HSV = ([25, 40, 25], [115, 255, 255])  # מכסה גם ציאן וגם ירוק
CYAN_HSV = ([78, 70, 25], [115, 255, 255])  # חצי שמאלי
GREEN_HSV = ([25, 70, 25], [76, 255, 255])  # חצי ימני
```

- **טווח רחב (H:25-115):** סובלני להיסט צבע מזווית צפייה, השתקפויות זכוכית ועיבוד מצלמה
- **רוויה ≥40 (רחב) / ≥70 (פיצול):** סף רחב יותר לזיהוי, הדוק יותר לכיווניות מבוססת צבע
- **ערך ≥25:** מזהה סמן גם בבהירות מסך נמוכה או בסביבות מעומעמות

## 4. תהליך הזיהוי (Detection Pipeline)

### שלב 1: מיסוך צבע HSV
```
קלט: פריים RGB מהמצלמה
→ המרה למרחב צבעים HSV
→ החלת inRange() עם פרמטרי WIDE_HSV
→ פעולות מורפולוגיות:
   - Open (אליפסה 3×3): הסרת רעש
   - Close (אליפסה 5×5): מילוי פערים
פלט: מסכה בינארית של פיקסלים ירוקים/ציאן
```

**מטרה:** בידוד כל הפיקסלים שיכולים להשתייך לסמן, דחיית גווני עור, קירות, רהיטים.

### שלב 2: זיהוי קווי מתאר (Contours) וסינון מלבניות
```
קלט: מסכה בינארית
→ findContours() לחילוץ אזורים מחוברים
→ לכל קו מתאר:
   - חישוב minAreaRect() (מוצא מלבן תוחם מינימלי)
   - חישוב מלבניות = area_contour / area_rect
   - דחייה אם מלבניות < 0.45 (לא סדיר מדי)
   - דחייה אם יחס גובה-רוחב < 0.03 (פס דק, ככל הנראה ארטיפקט)
פלט: בועות מלבניות מועמדות
```

**למה minAreaRect?** הוא מטפל בסיבוב באופן טבעי—אין צורך בתיבות תוחמות מיושרות לצירים.

### שלב 3: אימות פיצול דו-צבעי
```
לכל מועמד מלבני:
→ חילוץ פיקסלים בתוך תיבה מסובבת
→ החלת מסנן CYAN_HSV → ספירת פיקסלים ציאן
→ החלת מסנן GREEN_HSV → ספירת פיקסלים ירוקים
→ דרישה: פיקסלים_ציאן > 12% וגם פיקסלים_ירוקים > 12% משטח כולל
```

**מטרה:** הבחנה בין הסמן שלנו לבין עצמים ירוקים אקראיים (צמחים, ביגוד, פוסטרים). מלבן ירוק חד-צבעי נדחה.

### שלב 4: בדיקת סביבה כהה (גיבוי)
```
אם פיצול דו-צבעי נכשל (למשל, בזוויות קיצוניות עם ערבוב צבעים):
→ הרחבת מסכת המסך באליפסה 9×9
→ חילוץ טבעת = מורחב − מקורי
→ השוואה: median(בהירות_טבעת) < median(בהירות_מסך) × 0.75
```

**מטרה:** מסך OLED תמיד בהיר יותר מהסביבה (פרק יד/ביגוד), גם אם הצבעים מתערבבים.

### שלב 5: סידור פינות (Corner Ordering)
**אתגר:** `minAreaRect()` מחזיר 4 פינות בסדר שרירותי. להערכת תנוחה, אנחנו צריכים תיוג עקבי: [TL, TR, BR, BL].

**שיטה עיקרית - מרכזי צבע:**
```
→ חישוב מרכז כובד של כל הפיקסלים הציאן → cyan_center
→ חישוב מרכז כובד של כל הפיקסלים הירוקים → green_center
→ לכל פינה:
   - אם קרובה יותר ל-cyan_center → צד שמאלי (TL או BL)
   - אם קרובה יותר ל-green_center → צד ימני (TR או BR)
→ בתוך כל צד, מיון לפי קואורדינטת Y (עליון ראשון)
→ תוצאה: [TL_ציאן, TR_ירוק, BR_ירוק, BL_ציאן]
```

**גיבוי - סידור גאומטרי:**
```
אם פיצול צבע לא זמין:
→ מיון 4 פינות לפי Y → פיצול לזוג עליון וזוג תחתון
→ בתוך כל זוג, מיון לפי X (שמאלי ראשון)
```

### שלב 6: אימות קמירות (Convexity)
**בעיה:** פינות שסודרו לא נכון יוצרות "עניבת פרפר" (bowtie - מרובע חוצה), גורמות לכשלים קטסטרופליים ב-solvePnP.

**פתרון:**
```python
def _is_convex(corners):
    # כל המכפלות הוקטוריות של צלעות רצופות חייבות להיות באותו סימן
    for i in range(4):
        edge1 = corners[(i+1)%4] - corners[i]
        edge2 = corners[(i+2)%4] - corners[(i+1)%4]
        cross = edge1[0]*edge2[1] - edge1[1]*edge2[0]
        if sign(cross) שונה מהקודם → לא קמור
    return True
```

אם לא קמור, נסה 3 תמורות חלופיות של פינות עד שנמצא סידור קמור.

### שלב 7: אכיפת עקביות פינות
**בעיה:** סידור פינות בין פריימים יכול להתהפך עקב רעש במרכזי הצבע, גורם לריצוד.

**פתרון:**
```
אם פינות הפריים הקודם ידועות:
→ חישוב מרחק בין new[i] ל-prev[i] לכל i
→ אם max_distance > 80px → יתכן שפינות החליפו מקומות
→ הקצאה מחדש של פינות חדשות בהתאם ל-nearest-neighbor של prev
```

זה יוצר רציפות זמנית, מונע התהפכויות פתאומיות של כיוון.

## 5. הערכת תנוחה (6DOF Tracking)

### אלגוריתם Perspective-n-Point (PnP)

**קלט:**
- **מודל 3D:** ארבע פינות של מסך השעון הפיזי במילימטרים
  ```
  TL: (0, 0, 0)
  TR: (37.6, 0, 0)      # רוחב שעון 45mm
  BR: (37.6, 46.0, 0)   # גובה שעון 45mm
  BL: (0, 46.0, 0)
  ```
- **נקודות תמונה 2D:** פינות שזוהו בקואורדינטות פיקסלים
- **מטריצת מצלמה:** מוערכת מגודל הפריים (אורך מוקד ≈ 0.85 × רוחב)

**אלגוריתם:** `solvePnP` של OpenCV עם דגל `SOLVEPNP_IPPE_SQUARE`
- IPPE (Infinitesimal Plane-based Pose Estimation) מותאם למלבנים מישוריים
- מחזיר וקטור סיבוב (`rvec`) ווקטור תזוזה (`tvec`)

**פלט:**
- `tvec`: מיקום 3D של מוצא הסמן בקואורדינטות מצלמה (מ"מ)
- `rvec`: ייצוג ציר-זווית של הסיבוב

### בדיקת שגיאת הקרנה חוזרת (Reprojection Error)
**מטרה:** זיהוי פתרונות PnP גרועים שנגרמים מפינות שסודרו לא נכון או רועשות.

```python
# הקרנת המודל 3D חזרה ל-2D באמצעות התנוחה המוערכת
reprojected_corners = projectPoints(model_3D, rvec, tvec, camera_matrix)

# חישוב מרחק ממוצע בין פינות מוקרנות לפינות שזוהו
error = mean(||reprojected[i] - detected[i]||)

# דחייה אם השגיאה > פי 3 מאלכסון המסך (מעיד על bowtie או תנוחה הפוכה)
if error > 3 * ||corner[0] - corner[2]||:
    reject_pose()
```

### חישוב מרחק
```python
distance_mm = ||tvec|| = sqrt(tvec[0]² + tvec[1]² + tvec[2]²)
distance_cm = distance_mm / 10
```

### זווית סיבוב הגוף (Body Rotation)

**קונצפט:** הזווית בין וקטור הנורמל של מסך השעון לציר האופטי של המצלמה.

**מתמטיקה:**
```
1. המרת rvec למטריצת סיבוב R (3×3) באמצעות נוסחת Rodrigues
2. נורמל המסך במערכת המצלמה: n = R[:,2]  (עמודה שלישית של R)
3. ציר אופטי של המצלמה: z = [0, 0, 1]
4. זווית סיבוב הגוף: θ = arccos(|n·z|) = arccos(|n[2]|)
```

**פרשנות:**
- **0° = חזיתי:** מסך השעון פונה ישירות למצלמה (מיקום V1-V2 ב-ECG)
- **30-50° = צדדי:** מסך מוטה (מיקום V3-V4)
- **70-80° = צדדי רחוק:** הטיה קיצונית (מיקום V5-V6)

**למה abs()?** אנחנו מתעניינים רק בגודל ההטיה, לא בכיוון הנורמל (פנימה או החוצה).

### מיפוי מיקומי ECG
```python
if body_rotation < 15°:  "V1-V2 (חזיתי)"
elif body_rotation < 35°: "V3 (אמצע)"
elif body_rotation < 55°: "V4 (אמצע-צדדי)"
elif body_rotation < 70°: "V5 (צדדי)"
else:                     "V6 (צדדי רחוק)"
```

## 6. החלקה זמנית ויציבות

### האתגר
פלט הגולמי של solvePnP רועש עקב:
- שגיאות זיהוי תת-פיקסליות במיקומי פינות
- רעש חיישן המצלמה
- שינויי תאורה עדינים
- קוונטיזציה בסף HSV

### פתרון 1: ממוצע נע אקספוננציאלי (EMA)

**עבור תזוזה (מיקום):**
```python
tvec_smoothed = α * tvec_previous + (1-α) * tvec_current
כאשר α = 0.55 (מקדם החלקה)
```

**עבור סיבוב (כיוון):**
לא ניתן פשוט לממצע וקטורי סיבוב (מרחב לא-לינארי). במקום:
```python
1. המרת גם rvec_prev וגם rvec_curr למטריצות סיבוב
2. עירוב: R_blend = α * R_prev + (1-α) * R_curr
3. אורתוגונליזציה מחדש באמצעות SVD: R_blend = U @ V^T
4. וידוא det(R) = +1 (סיבוב תקין)
5. המרה חזרה ל-rvec
```

זהו SLERP (Spherical Linear Interpolation) מפושט למטריצות סיבוב.

### פתרון 2: דחיית חריגים (Outliers)

**זיהוי קפיצות:**
```python
if |distance_current - distance_previous| / distance_previous > 0.6:
    # ככל הנראה פריים גרוע → העלאת החלקה ל-α = 0.85
    # (העדפה כבדה להערכה הקודמת)
```

מונע מזיהויים שגויים חולפים לגרום לקפיצות פראיות.

### פתרון 3: מעקב ROI

**אופטימיזציה:** לאחר זיהוי הסמן, חפש רק באזור סביב המיקום הקודם.

```python
roi = (x - expand*width, y - expand*height, 
       x + expand*width, y + expand*height)
```

- מפחית false positives מעצמים רחוקים
- משפר FPS על ידי עיבוד פחות פיקסלים
- נסיגה לחיפוש מסגרת מלאה אם המעקב אבד ל->25 פריימים

### פתרון 4: זיכרון פינות

שמירת `_prev_corners` ואכיפת עקביות באמצעות nearest-neighbor matching (מתואר בסעיף 4, שלב 7).

## 7. מצבי כשל והפחתתם

| **מצב כשל** | **גורם** | **הפחתה** |
|-------------|----------|-----------|
| **False Positive (עצם ירוק אקראי)** | ביגוד ירוק, צמחים, פוסטרים | אימות פיצול דו-צבעי: דרישה לגם ציאן וגם ירוק |
| **OLED Bloom** | בהירות 100% גורמת לדליפת פיקסלים | הפחתת בהירות ל-60%; טווח HSV רחב סובלן להיסט צבע |
| **השתקפות זכוכית** | הדגשות ספקולריות מוסיפות אור לבן | סף רוויה ≥40 ב-HSV רחב; השתקפויות מורידות רוויה אך לא מבטלות גוון |
| **פינות חוצות (bowtie)** | הקצאת פינות לא נכונה | בדיקת קמירות + סידור מחדש אוטומטי דרך החלפות |
| **ריצוד סידור פינות** | מרכזי צבע נעים בין פריימים | התאמת nearest-neighbor לפינות הפריים הקודם |
| **הטיה קיצונית (V6 ב-75°)** | המסך מתכווץ לרצועה דקה | סמן מלא-מסך מספק פי 100 יותר פיקסלים מנקודות |
| **פתרון solvePnP גרוע** | פינות רועשות או כמעט מישוריות | בדיקת שגיאת הקרנה חוזרת: דחה אם error > פי 3 מהאלכסון |
| **פלט מרצד** | רעש מצלמה, קוונטיזציה | החלקה אקספוננציאלית (α=0.55) + דחיית חריגים |

## 12. נספח: פסאודו-קוד אלגוריתם

```python
# לולאת זיהוי ברמה גבוהה
while camera.is_open():
    frame = camera.read()
    
    # שלב 1: פילוח צבע
    hsv = cvtColor(frame, BGR2HSV)
    mask = inRange(hsv, WIDE_HSV_MIN, WIDE_HSV_MAX)
    mask = morphologyEx(mask, OPEN, kernel=3×3)
    mask = morphologyEx(mask, CLOSE, kernel=5×5)
    
    # שלב 2: מציאת בועות מלבניות
    contours = findContours(mask)
    candidates = []
    for contour in contours:
        rect = minAreaRect(contour)
        if rectangularity(contour, rect) < 0.45:
            continue
        
        # שלב 3: אימות פיצול דו-צבעי
        box = boxPoints(rect)
        cyan_pct, green_pct = measure_color_split(hsv, box)
        if cyan_pct > 0.12 and green_pct > 0.12:
            candidates.append((rect, box, score=3.0))
        elif has_dark_surround(frame, box):
            candidates.append((rect, box, score=1.5))
    
    if not candidates:
        display("מחפש סמן...")
        continue
    
    # שלב 4: בחירת מועמד הטוב ביותר
    best = max(candidates, key=lambda c: c.score * c.rect.area)
    corners = best.box
    
    # שלב 5: סידור פינות
    corners = order_by_color_centroids(corners, hsv)
    corners = fix_if_crossed(corners)  # בדיקת קמירות
    corners = match_to_previous(corners, prev_corners)
    
    # שלב 6: הערכת תנוחה
    ok, rvec, tvec = solvePnP(model_3D, corners, camera_matrix)
    if not ok:
        continue
    
    # שלב 7: אימות והחלקה
    reproj_error = compute_reprojection_error(rvec, tvec, corners)
    if reproj_error > 3 * diagonal(corners):
        reject()
    
    tvec = smooth(tvec, prev_tvec, α=0.55)
    rvec = smooth_rotation(rvec, prev_rvec, α=0.55)
    
    # שלב 8: חישוב סיבוב גוף
    R = rodrigues(rvec)
    normal = R[:, 2]  # וקטור נורמל המסך
    body_rotation = arccos(|normal[2]|) * 180/π
    
    # שלב 9: תצוגת תוצאות
    draw_axes(frame, rvec, tvec)
    draw_info(frame, distance=||tvec||, body_rotation)
    draw_corners(frame, corners)
    show(frame)
```
